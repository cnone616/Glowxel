# 开发强约束与回归检查

本文件用于约束后续在 `uniapp/` 与 `esp32-firmware/` 上的联动开发，重点防止以下问题再次发生：

- 为了“优化”而重写原本可用的核心链路
- 一次改动牵连多个无关模块，导致回归范围失控
- 连接、传输、模式切换、时间同步等基础能力被误伤
- 改完没有先自检，直接交给人工兜底测试

## 1. 适用范围

以下改动必须遵守本文件：

- `utils/webSocket.js`
- `store/device.js`
- `pages/control/control.vue`
- `pages/clock-editor/*`
- `pages/spirit-screen/*`
- `pages/canvas-editor/*`
- `esp32-firmware/src/main.cpp`
- `esp32-firmware/src/websocket_handler.cpp`
- `esp32-firmware/src/websocket_command_handlers.cpp`
- `esp32-firmware/src/wifi_manager.cpp`
- 所有会影响设备连接、模式切换、素材发送、时间同步、屏幕渲染的文件

## 2. 需要先确认的改动

### 2.0 已定板模式保护

以下模式视为当前项目的已完成成品，后续不得拿来做试验性重构、借鉴仓库替换或顺手改造：

- `桌面宠物`
- `静态时钟`
- `动态时钟`
- `主题模式`
- `画板模式`

以下改动不是绝对不能做，但在动手前必须先向用户确认：

- 重写 `WS` 连接流程。
- 把“局部问题修复”扩展成页面结构重做、状态流重做或协议重做。
- 新增、删除、重命名连接态相关状态字段。
- 改动会影响现有显示逻辑的模板结构调整。
- 改动会影响现有设备协议、命令顺序、素材发送顺序的链路重排。
- 改动会影响板载程序主循环调度方式、模式分支职责或后台任务触发位置的调整。

以下仍然属于禁止事项：

- 禁止把单一状态拆成两份来源并行维护。
  - 例如页面本地状态和 `store` 状态同时代表“是否已连接”。
- 禁止把命令超时视为连接断开。
  - `等待回复超时` 只能表示当前命令失败，不能顺手把 `connected` 置为 `false`。
- 禁止把某项后台能力只挂在单一模式分支下。
  - 例如时间同步只在 `clock + 未连接客户端` 时重试。
- 禁止在未完成自检前要求人工先验证基础链路。
- 禁止“顺手优化”核心链路周边代码。
  - 如果用户要修连接问题，就只修连接问题；若要顺带改布局、提示文案、状态管理结构，必须先确认。

## 3. 改动前强制检查

在修改核心链路前，必须先逐条确认：

1. 这次改动是否只解决当前明确问题。
2. 是否会影响以下既有能力：
   - 连接设备
   - 获取设备状态
   - 模式切换
   - 像素/动画二进制发送
   - 时间同步
   - 设备不断线保持
3. 是否存在“更小的改法”。
4. 是否准备复用旧实现，而不是直接重写。
5. 是否把状态来源收敛为单一真值。
6. 如果这次改动超出了当前明确问题，是否已经先得到用户确认。

如果以上任一项不能明确回答，停止编码，先补充分析。

## 4. 核心链路改动规则

### 4.1 WebSocket

- 连接状态只允许一个主来源。
- `onError` 不得直接等同于 `onDisconnect`。
- 自动重连只能响应真实断开，不得因为普通命令失败触发。
- 改动 `connect / onOpen / onClose / onError / sendAndWait / getStatus` 时，必须检查整条连接生命周期。

### 4.2 设备控制页

- 已连接态与未连接态的显示必须只由真实连接状态决定。
- 连接入口、配网入口、断开按钮的展示条件必须可一眼读懂。
- 热点配网入口与说明页必须明确提示“仅支持 2.4GHz WiFi”，且主路径固定为“连接设备热点后打开 `192.168.4.1`”。
- 热点配网页不得直接套用粗糙默认门户样式；若借鉴外部实现，只允许复用其配网模式与扫描行为，不允许跳过本项目既有信息层级与提示结构。
- 配网链路若声明参考外部来源，必须能落到已确认源码路径；本项目当前热点配网参考源固定为 `clockwise` 的 AP 配网行为，不允许再拍脑袋补另一套交互。
- 不允许为了隐藏某个元素而顺手重构整页模板，除非用户明确确认。
- “设备参数”必须作为独立页面入口接入设备控制，不允许顺手改坏原有展示模式入口与页面主流程。
- 样式预览类总览页如果作为联调入口接入，必须从设备控制页可直接进入，且保持离线可直达，不允许再额外藏到深层调试路径里。
- 设备参数页只承载核心开关设置，不允许借接入参数页之机改动既有时钟、主题、动画、画板的展示逻辑。
- 设备参数页优先展示用户当前能理解、能直接解决兼容问题的参数；不允许一次性把教程里所有底层术语堆进页面，导致参数层失焦。
- 当前设备参数页的主重点固定为：亮度、旋转、色序、相位、驱动兼容、NTP，以及少量可理解的设备信息；像时区/POSIX/LDR 这类策略项未确认前不得强塞进首屏。
- 设备参数页里的参数文案必须优先使用通俗中文，不允许把 `driver / i2cSpeed / E_pin / nightStart / ntpServer` 这类底层字段名直接当主标题丢给用户看。
- 设备参数页中的输入框、时间选择框、下拉框、只读值盒都必须有清晰可见的描边，继续沿用当前白底黑边的视觉体系，不能做成无边框弱控件。
- 当用户在设备参数页只点名少数控件的边框或尺寸问题时，改动范围只允许落在该页面对应控件本身，不允许顺手扩到其他页面、其他参数块或接口逻辑。
- 设备参数页不再作为离线可浏览页面；进入时或停留期间只要设备未连接，必须立即退出当前页并返回设备控制页。
- 设备控制页里的“设备参数”入口文案必须用通俗中文描述用户能直接理解的内容，不再把 `特殊 LED / 驱动 / E Pin` 这类术语直接挂在入口说明里。
- 设备控制页点击 `设备参数` 入口时，必须先判断真实连接状态；未连接时不允许先跳进页面再退回，直接在当前页拦截即可。
- 设备控制页里这类“请先连接设备”的拦截提示，必须绑定项目现有 `Toast` 组件显示，不允许退回到原生 `uni.showToast` 视觉。
- 全仓页面里的通知提示、确认操作、加载中流程统一走项目自定义 `Toast / Confirm / Loading` 组件链路；除工具封装内部兜底外，页面和业务组件里不允许直接写 `uni.showToast / uni.showModal / uni.showLoading`
- 亮度语义保持既有约定：UI 仍使用 `0-100`，下发设备时继续映射到 `0-178`，不得私自改成满量程 `255` 直通。
- 面板兼容项接入时，字段名必须沿用已确认来源，不允许再造别名字段或模糊开关字段。
- LED 兼容设置页必须支持“颜色下拉 + 相位开关”的组合交互，其中颜色下拉只作为 UI 表达，底层继续映射到已确认的显示兼容来源，不允许再造模糊色序字段。
- NTP 设置必须支持手动切换 `ntpServer`，默认值要优先国内可用地址，不允许继续把国外默认地址作为首选。

### 4.3 板载程序主循环

- 后台任务不能依赖单一模式分支才能运行。
- 连接成功后需要持续生效的任务必须全局调度。
  - 例如时间同步、超时清理、后台重试。
- 对 `main.cpp` 的修改必须额外检查：
  - `MODE_CLOCK`
  - `MODE_THEME`
  - `MODE_ANIMATION`
  - `MODE_CANVAS`
  - 客户端已连接
  - 客户端未连接

### 4.4 素材与像素发送

- 修复发送链路时，禁止改坏既有二进制发送方式。
- 命令发送顺序调整前，必须确认不会阻塞后续二进制数据。
- 涉及前后端协议时，优先复用现有字段与流程，不允许臆造兼容分支。

### 4.5 一级页底部导航与切换稳定性

- 用户已最新纠正：`首页 / 社区 / 创作 / 设备 / 我的` 5 个一级页当前继续使用原生 `tabBar`，页面内自定义底部导航方案先暂停，不再作为当前实现方向。
- 只要自定义底部导航仍会触发页面横向偏移、跟页抖动、切换错位或肉眼可见的不稳定感，就不得继续强推该方案。
- 一级页路由与缓存能力继续沿用现有原生 `tabBar + switchTab` 链路，不允许为试验自定义底栏顺手改成另一套跳转协议。
- 后续若再次尝试自定义底部导航，前提必须是先独立查清“页面宿主偏移/切换抖动”根因；未定位根因前，不得再次接入正式一级页。
- 当前阶段关于一级页底部导航的默认策略是“优先稳定，不为换实现方式引入新问题”。

## 5. 强制自检清单

只要改动了核心链路，交付前必须至少完成以下自检：

### 5.1 前端自检

- 未连接时：
  - 应显示连接入口
  - 不应显示仅连接后可用的控制区
- 已连接时：
  - 不应继续显示 `WiFi 连接 / 蓝牙配网`
  - 模式入口、亮度、高级操作可见
- 命令超时时：
  - 只报当前命令失败
  - 不应自动把连接态切成未连接

### 5.2 固件自检

- `pio run` 必须通过
- 若改动 `main.cpp / wifi_manager.cpp / websocket_*`：
  - 检查不同模式下是否还能进入主循环
  - 检查后台任务是否仍会执行
- 若改动渲染逻辑：
  - 检查是否引入清屏式闪烁
  - 检查是否仍保留差分更新路径
  - 单缓冲设备上，`planet_screensaver` 与 `ambient_effect / led_matrix_showcase` 这类全屏动画必须走“离屏绘制 -> 一次提交”链路，禁止在真实显示缓冲上先 `fillScreen` 再逐像素覆盖
  - 新增全屏动画场景时，必须验证“桌宠稳定、星球稳定、新场景稳定”三类观感一致，不允许出现“桌宠不闪但其他模式频闪”的提交结果

### 5.3 联动自检

- 连接成功后能收到状态
- 切模式不会误断开
- 发送素材后不会因为等待回复超时而中断整条链路
- 后台任务不会因为当前模式变化而停摆

2. 再写入对应约束文档

### 8.0.2 复刻场景协议与参数规则（新增）

用户已明确确认：GitHub 参考迁入的场景不能只复刻一个“看着像”的预览，协议字段、前端参数、固件渲染语义也必须同步对齐。

执行规则：

- 复刻类场景新增参数时，必须先把协议字段和页面参数一一对齐，不能前端一套、固件一套
- 不允许继续保留已经被用户否定的旧参数语义，再用文案包装成新效果
- 某个复刻场景如果已经确认参数集合，后续该场景的发送链路、预览链路、固件解析链路都必须使用同一组字段
- 若只升级单个场景，必须把该场景相关入口全部同步，不允许同一场景在不同页面出现一处新参数、一处旧参数

本轮雨幕场景已确认的链路规则：

- 雨幕参数按 `preset / speed / density / color / loop` 对齐
- 雨幕渲染不允许再使用旧的拖痕式尾迹表现

### 8.0.1 预览网格线硬约束（新增）

用户已明确确认：所有功能页预览区域必须可见 `64×64` 网格线，用于判断预览尺寸与设备尺寸是否一致。

执行规则：

- 只要是功能预览画布（非最终设备实拍图），默认必须显示网格线
- 网格线必须与 `64×64` 像素坐标一致，不允许出现非等距或错位
- 不允许在后续优化中私自关闭网格线或改成“默认隐藏”
- 若存在“编辑器手绘场景”这类确需切换网格的页面，默认仍需开启，且不得影响功能页预览默认显示

### 8.1 设计输入优先级

当用户提供以下任一设计参考时，必须先分析参考，再写代码：

- Figma 链接或导出图
- 页面截图、参考网站、竞品页面
- 已有项目内成熟页面
- 设计文档、风格说明、图标规范

禁止行为：

- 未分析参考就凭经验直接改样式
- 只参考“大概感觉”而不提取具体数值
- 把“参考某页”理解成“允许自由重设计”

如果设计信息不完整，必须先提出具体问题，不能自行脑补：

- 颜色值不明确
- 字号、字重、行高不明确
- 间距、圆角、阴影、描边规则不明确
- 图标风格、插画风格、动效风格不明确

### 8.2 样式实现规则

#### 8.2.1 精确值优先

涉及以下内容时，优先使用参考中可确认的精确值，不允许近似替代：

- 颜色
- 字号
- 字重
- 行高
- 间距
- 组件高度
- 圆角
- 描边
- 阴影

禁止写法：

- “看起来差不多”的近似值替换
- 没有来源的随机新颜色
- 无依据修改已有字号体系

#### 8.2.2 交互反馈禁令

后续所有页面、组件、弹窗、卡片、入口块、tabs、按钮、返回按钮、选项块，统一禁止以下按压反馈：

- `scale`
- `translate`
- `rotate`
- 抖动
- 阴影跳变
- 任何“点击后位置偏移一下”的动效

允许的反馈只有：

- 颜色变化
- 边框变化
- 字重变化
- 内容切换本身

执行要求：

- 不允许再写 `:active { transform: ... }`
- 不允许通过 `transition` 包装按压位移形成“假装不明显”的摇晃效果
- 如果历史页面存在上述写法，优先从公共样式层和公共组件层清掉，再处理页面残留

#### 8.2.2 设计 Token 优先

涉及全局或跨页面复用的视觉值时，必须优先收敛到统一 token 或现有语义变量：

- 文本颜色
- 背景颜色
- 边框颜色
- 主色与状态色
- 圆角等级
- 间距等级
- 阴影等级

禁止行为：

- 同一语义在不同页面各写一套值
- 新页面绕开已有主题变量另起体系
- 为局部页面临时发明一组不可复用的样式常量

#### 8.2.3 组件一致性优先

按钮、卡片、输入框、导航、标签、弹层、提示、图标容器等通用组件，必须遵守一致性：

- 同层级组件的圆角、内边距、字号、触感反馈要统一
- 同一功能语义在不同页面视觉表达应稳定
- 不允许一个页面是极简白色，另一个页面突然变成重装饰卡片风

若已有成熟页面可复用，必须先向成熟页面对齐，而不是另做一版。

补充强约束：

- `功能配置页` 属于同一设计域，不能只改一个页面而放任同组页面继续使用旧结构或旧样式
- 如果用户指出“这一类页面要统一”，后续不得再以“本次只改当前页”为理由跳过同组页面排查
- 只要改到页面公共骨架、顶部导航、滚动容器、搜索栏、tabs 栏，就必须全量排查所有使用同类结构的页面
- 任何带顶部导航的页面都必须保证“顶部固定 + 内容区独立滚动”，不能让页面整体滚动把标题栏顶走

#### 8.2.4 布局与信息密度约束

布局改动必须同时满足：

- 结构清晰，主次明确
- 间距有规律，不随手写零散数值
- 不为“显得完整”而重复展示同一信息
- 不为“显得高级”而堆叠副标题、说明文、装饰块

禁止行为：

- 一个区域内重复出现同一状态或同一功能说明
- 无必要的辅助文案堆叠
- 无意义留白或无意义拥挤
- 仅为了美化而增加用户不需要的视觉层

#### 8.2.4.1 离线浏览约束

当项目处于联调、修复 `WS`、排查设备链路或纯样式调整阶段时：

- 设备控制页必须允许用户在未连接设备时直接进入各功能页面查看样式
- 功能入口、亮度区、设置区可以离线展示，不允许因为未连接就整块隐藏
- 只有真正依赖设备的发送、重置、同步、下发类动作才保留未连接拦截
- 不允许把“样式预览”和“设备联动”绑成同一个前置条件

#### 8.2.5 动效与反馈约束

动效必须服务于状态表达和操作反馈，不得为了“有动效”而动：

- 页面切换、加载、hover、press、toast、弹层等反馈要统一
- 时长、缓动、透明度变化要克制
- 不允许引入与硬件显示冲突的频闪式视觉表达

禁止行为：

- 大面积频繁闪烁
- 无业务意义的装饰性动效
- 同类交互在不同页面使用完全不同的反馈节奏

### 8.3 项目视觉基调补充约束

结合当前项目实际，后续默认遵循以下风格基线：

- 白色为主的极简主义
- 克制、清爽、留白明确
- 强调功能可读性，不靠花哨装饰制造“设计感”
- 图标可有设计感，但不能破坏整体简洁基调

在未得到用户明确确认前，禁止：

- 把页面改成深色主导
- 引入大面积渐变、玻璃拟态、厚重阴影
- 大幅改变成熟页面既有结构
- 为追求“更丰富”而加无关说明、徽章、说明卡片

### 8.4 设计保真工作流

当任务属于“还原设计稿/对齐参考图/替换为新设计”时，必须按以下顺序执行：

1. 先分析参考来源
2. 提取视觉规格
3. 明确哪些内容来自设计稿，哪些是基于现有项目推断
4. 再开始写样式或改结构
5. 交付前做一次保真核对

提取视觉规格时，至少覆盖：

- 颜色系统
- 字体系统
- 间距系统
- 组件规格
- 布局结构
- 阴影/描边/圆角
- 动效与反馈
- 图标风格

### 8.5 保真与回归自检

只要改动了页面视觉或样式，交付前至少完成以下检查：

- 是否与用户给的参考一致，而不是“我觉得更好”
- 是否与项目现有白色极简风格一致
- 是否复用了已有 token、变量或成熟组件样式
- 是否引入了无来源的新颜色、新字号、新圆角体系
- 是否造成了同类组件风格分裂
- 是否增加了无意义文案、重复信息、重复入口
- 是否影响了触控区域、可读性与状态辨识

### 8.6 文档维护规则

当用户后续再补充新的设计规范、Figma 输出规则、图标要求、页面风格禁令时，应优先更新本文件，而不是只停留在对话里。

补充执行要求：

- 同一条口头约束如果用户已经重复强调超过一次，必须视为高优先级硬规则

- 不允许继续依赖“对话里记着就行”的方式执行，否则视为流程不合格

- 后续样式调整必须以“保持已确认约束稳定”为前提，禁止一边修一边破坏之前已定规则
- 例如用户已明确否定左侧选中色条、否定“安全区和额外 padding 叠加导致的过高 tabs 留白”，后续不得再次引入

- 例如用户已明确要求俄罗斯方块时钟不要再和旧版俄罗斯方块屏保玩法混搭，后续不得再次把旧自定义控制面板拼回该页面

- `Universal-LPC-Spritesheet-Character-Generator` 已确认不适合当前项目，不再保留本地接入、脚本、目录或后续中文化维护；若未来要再次引入，必须先重新征得用户确认

- 项目确认新的颜色/字体/间距体系
- 项目确认新的图标导出与使用规范
- 项目确认新的页面层级与信息密度规则
- 项目确认新的动效限制与交互反馈标准

### 8.7 代理执行质量门禁（新增）

用户已新增负面反馈并确认：当前代理在分析与执行阶段存在“偷工减料、频繁让用户补步骤、无法真正省时省力”的问题；后续必须按以下门禁执行。

执行硬约束：

- 默认按“端到端交付”执行：拿到目标后，除非遇到真实阻断（权限、外部账号、信息缺失且无法推断），不得把中间步骤转嫁给用户。
- 默认按“少打扰”执行：可自行完成的信息收集、排查、实现、回归，必须先做完再汇报，不得把可由代理完成的动作反向要求用户执行。
- 默认按“结果导向”汇报：输出必须包含“改了什么、为什么这样改、怎么验证、还有什么风险”，禁止只给思路不落地。
- 未验证项必须显式披露：凡是未跑的测试、未做的实机验证、无法完成的校验，必须明确写出原因和影响，禁止省略。
- 高频问题必须一次性处理：若同类问题已被用户重复指出，不允许再以“先这样、后续再看”拖延，必须本轮收口并给出可核对结果。
- 阻断问题最小化提问：确需用户决策时，只允许提出最少且必要的问题，并同时给出可执行建议与影响，不使用开放式泛问。
- 连续执行场景下（用户已表达希望省心/省时），不得在非关键节点反复等待确认；应在安全边界内持续推进直到可验收结果。
- 对于“素材检索 / 资料归档 / 截图比对 / 资源整理”这类任务，若用户明确要求“赶紧整出来”，默认直接产出本地整理结果、下载成果和目录落地，不再用长篇过程说明代替交付。

交付前自检清单（每次任务结束前必须自查）：

- 是否已经完成需求对应的代码落地，而不是只停留在分析或计划。
- 是否已完成至少一轮可执行验证（脚本、构建、静态检查、或明确说明为何无法验证）。
- 是否把可由代理完成的步骤留给了用户手工处理；若有，是否属于真实阻断。
- 是否存在“偷工减料”信号：只改表层、不做回归、不覆盖关键路径、避开用户明确痛点。
- 是否给出了下一步最小行动方案，且不增加用户理解与操作负担。

### 8.8 新建页图片预览删色替色门禁（新增）

用户已于 2026-04-28 最新确认：本轮“使用的颜色”交互只允许收口在新建页预览阶段，不得扩散到编辑页、项目长期存储合同或其他模式页面。

执行硬约束：

- 作用范围只限 `pages/create/*` 当前预览链；不得顺手修改编辑页、项目 `palette` 长期合同或其他页面的颜色语义。
- 新建页预览中的“使用的颜色”必须允许用户处理单个已用色。
- 当用户删除某个已用色时，不能直接留空；必须把该颜色对应像素替换成当前已选套装中的其他颜色。
- 自动删除路径必须执行“就近取色替换”；替换来源固定为当前已选套装中的其他颜色。
- 同时允许用户手动指定替换色；手动替换候选来源固定为“当前已选套装的全部颜色”，与该套装当前选中的颜色种类数保持一致。
- 删色 / 替色只修改当前创建预览结果，不承担“创建完成后继续按原图或按历史设置重算”的职责。

### 8.9 发送动作占位快照与 Toast 显示门禁（新增）

用户已于 2026-04-28 最新确认：各模式页在未连接板载程序 WebSocket 或发送未收到响应时，点击发送后的提示反馈也不能被预览 Canvas、播放层或其它内容区遮挡。

执行硬约束：

- 凡是页面存在“发送到设备 / 推送 / 应用 / 同步”这类发送动作，触发点击后就必须立即进入统一的发送期前置反馈链路；不能把占位快照、遮罩或层级切换依赖在 WebSocket 成功响应之后。
- 即使当前未连接板载程序 WebSocket，发送失败类 Toast 也必须完整显示在最上层，不能被 Canvas、预览图层、占位图层或页面局部 fixed 区块遮住。
- 若页面使用“发送中占位快照 / 预览冻结图 / Canvas 替身图层”，其触发时机必须绑定在“发送动作开始”而不是“设备返回成功”。
- 修复范围优先落在发送公共链路与 Toast / 占位快照联动层，不允许为了单页现象把各模式页各自写一套临时遮挡补丁。

### 8.10 模式页时间 / 字体分组语义门禁（新增）

用户已于 2026-04-28 最新确认：模式页里 `时间` 与 `字体` tabs 的控件归属必须按语义拆分，不能把时间显示开关类字段塞进字体分组里。

执行硬约束：

- `显示秒钟` 属于时间显示语义，必须放在 `时间` tab；不得继续放在 `字体` tab 或字体选择组件内。
- `字体` tab 只承载字体选择本身及与字体样式直接相关的内容，不承担时间显示开关语义。
- 若多个模式页复用同一套 `时间 / 字体` 组件，修复时必须先识别共享边界，优先走最小公共改动，避免单页和公共组件出现两套不同分组语义。
- `snake-mode` 当前没有独立 `时间` tab 时，`显示秒钟` 必须放进 `参数` tab，不得继续留在 `外观` / 字体分组里。
- `snake-mode` 的 `显示秒钟` 放进 `参数` tab 后，必须复用项目现有 `GlxSwitch` 开关样式；不得误改成 `关闭 / 显示` 双按钮。
