# 开发强约束与回归检查

本文件用于约束后续在 `uniapp/` 与 `esp32-firmware/` 上的联动开发，重点防止以下问题再次发生：

- 为了“优化”而重写原本可用的核心链路
- 一次改动牵连多个无关模块，导致回归范围失控
- 连接、传输、模式切换、时间同步等基础能力被误伤
- 改完没有先自检，直接交给人工兜底测试

## 1. 适用范围

以下改动必须遵守本文件：

- `utils/webSocket.js`
- `store/device.js`
- `pages/control/control.vue`
- `pages/clock-editor/*`
- `pages/spirit-screen/*`
- `pages/canvas-editor/*`
- `esp32-firmware/src/main.cpp`
- `esp32-firmware/src/websocket_handler.cpp`
- `esp32-firmware/src/websocket_command_handlers.cpp`
- `esp32-firmware/src/wifi_manager.cpp`
- 所有会影响设备连接、模式切换、素材发送、时间同步、屏幕渲染的文件

## 2. 需要先确认的改动

以下改动不是绝对不能做，但在动手前必须先向用户确认：

- 重写 `WS` 连接流程。
- 把“局部问题修复”扩展成页面结构重做、状态流重做或协议重做。
- 新增、删除、重命名连接态相关状态字段。
- 改动会影响现有显示逻辑的模板结构调整。
- 改动会影响现有设备协议、命令顺序、素材发送顺序的链路重排。
- 改动会影响板载程序主循环调度方式、模式分支职责或后台任务触发位置的调整。

以下仍然属于禁止事项：

- 禁止把单一状态拆成两份来源并行维护。
  - 例如页面本地状态和 `store` 状态同时代表“是否已连接”。
- 禁止把命令超时视为连接断开。
  - `等待回复超时` 只能表示当前命令失败，不能顺手把 `connected` 置为 `false`。
- 禁止把某项后台能力只挂在单一模式分支下。
  - 例如时间同步只在 `clock + 未连接客户端` 时重试。
- 禁止在未完成自检前要求人工先验证基础链路。
- 禁止“顺手优化”核心链路周边代码。
  - 如果用户要修连接问题，就只修连接问题；若要顺带改布局、提示文案、状态管理结构，必须先确认。

## 3. 改动前强制检查

在修改核心链路前，必须先逐条确认：

1. 这次改动是否只解决当前明确问题。
2. 是否会影响以下既有能力：
   - 连接设备
   - 获取设备状态
   - 模式切换
   - 像素/动画二进制发送
   - 时间同步
   - 设备不断线保持
3. 是否存在“更小的改法”。
4. 是否准备复用旧实现，而不是直接重写。
5. 是否把状态来源收敛为单一真值。
6. 如果这次改动超出了当前明确问题，是否已经先得到用户确认。

如果以上任一项不能明确回答，停止编码，先补充分析。

## 4. 核心链路改动规则

### 4.1 WebSocket

- 连接状态只允许一个主来源。
- `onError` 不得直接等同于 `onDisconnect`。
- 自动重连只能响应真实断开，不得因为普通命令失败触发。
- 改动 `connect / onOpen / onClose / onError / sendAndWait / getStatus` 时，必须检查整条连接生命周期。

### 4.2 设备控制页

- 已连接态与未连接态的显示必须只由真实连接状态决定。
- 连接入口、配网入口、断开按钮的展示条件必须可一眼读懂。
- 不允许为了隐藏某个元素而顺手重构整页模板，除非用户明确确认。

### 4.3 板载程序主循环

- 后台任务不能依赖单一模式分支才能运行。
- 连接成功后需要持续生效的任务必须全局调度。
  - 例如时间同步、超时清理、后台重试。
- 对 `main.cpp` 的修改必须额外检查：
  - `MODE_CLOCK`
  - `MODE_THEME`
  - `MODE_ANIMATION`
  - `MODE_CANVAS`
  - 客户端已连接
  - 客户端未连接

### 4.4 素材与像素发送

- 修复发送链路时，禁止改坏既有二进制发送方式。
- 命令发送顺序调整前，必须确认不会阻塞后续二进制数据。
- 涉及前后端协议时，优先复用现有字段与流程，不允许臆造兼容分支。

## 5. 强制自检清单

只要改动了核心链路，交付前必须至少完成以下自检：

### 5.1 前端自检

- 未连接时：
  - 应显示连接入口
  - 不应显示仅连接后可用的控制区
- 已连接时：
  - 不应继续显示 `WiFi 连接 / 蓝牙配网`
  - 模式入口、亮度、高级操作可见
- 命令超时时：
  - 只报当前命令失败
  - 不应自动把连接态切成未连接

### 5.2 固件自检

- `pio run` 必须通过
- 若改动 `main.cpp / wifi_manager.cpp / websocket_*`：
  - 检查不同模式下是否还能进入主循环
  - 检查后台任务是否仍会执行
- 若改动渲染逻辑：
  - 检查是否引入清屏式闪烁
  - 检查是否仍保留差分更新路径

### 5.3 联动自检

- 连接成功后能收到状态
- 切模式不会误断开
- 发送素材后不会因为等待回复超时而中断整条链路
- 后台任务不会因为当前模式变化而停摆

## 6. 提交流程约束

- 一次只解决一个核心问题。
- 若必须连改多处文件，需保证这些文件服务于同一个问题。
- 若出现“顺手一起改更合理”的想法，必须先停下确认，不得自行扩大范围。
- 每次提交说明必须能回答：
  - 改了什么
  - 没改什么
  - 为什么这次不会误伤旧链路

## 7. 本文件的执行原则

- 本文件优先于“顺手优化”的冲动。
- 本文件优先于主观审美调整。
- 本文件优先于大范围重构。
- 如果用户要求的是修复，就按修复处理，不扩大成重设计。

## 8. UI 与样式强约束

本节基于新增的 `ui-fidelity-implementer.md` 与 `ui-design-architect.md` 提炼，用于约束后续所有页面视觉改动、组件样式调整、设计稿还原与视觉系统整理。

目标不是“自由发挥设计”，而是：

- 优先保证设计保真度
- 优先保证全项目视觉一致性
- 优先保证样式值可追溯、可复用、可验证

### 8.1 设计输入优先级

当用户提供以下任一设计参考时，必须先分析参考，再写代码：

- Figma 链接或导出图
- 页面截图、参考网站、竞品页面
- 已有项目内成熟页面
- 设计文档、风格说明、图标规范

禁止行为：

- 未分析参考就凭经验直接改样式
- 只参考“大概感觉”而不提取具体数值
- 把“参考某页”理解成“允许自由重设计”

如果设计信息不完整，必须先提出具体问题，不能自行脑补：

- 颜色值不明确
- 字号、字重、行高不明确
- 间距、圆角、阴影、描边规则不明确
- 图标风格、插画风格、动效风格不明确

### 8.2 样式实现规则

#### 8.2.1 精确值优先

涉及以下内容时，优先使用参考中可确认的精确值，不允许近似替代：

- 颜色
- 字号
- 字重
- 行高
- 间距
- 组件高度
- 圆角
- 描边
- 阴影

禁止写法：

- “看起来差不多”的近似值替换
- 没有来源的随机新颜色
- 无依据修改已有字号体系

#### 8.2.2 设计 Token 优先

涉及全局或跨页面复用的视觉值时，必须优先收敛到统一 token 或现有语义变量：

- 文本颜色
- 背景颜色
- 边框颜色
- 主色与状态色
- 圆角等级
- 间距等级
- 阴影等级

禁止行为：

- 同一语义在不同页面各写一套值
- 新页面绕开已有主题变量另起体系
- 为局部页面临时发明一组不可复用的样式常量

#### 8.2.3 组件一致性优先

按钮、卡片、输入框、导航、标签、弹层、提示、图标容器等通用组件，必须遵守一致性：

- 同层级组件的圆角、内边距、字号、触感反馈要统一
- 同一功能语义在不同页面视觉表达应稳定
- 不允许一个页面是极简白色，另一个页面突然变成重装饰卡片风

若已有成熟页面可复用，必须先向成熟页面对齐，而不是另做一版。

#### 8.2.4 布局与信息密度约束

布局改动必须同时满足：

- 结构清晰，主次明确
- 间距有规律，不随手写零散数值
- 不为“显得完整”而重复展示同一信息
- 不为“显得高级”而堆叠副标题、说明文、装饰块

禁止行为：

- 一个区域内重复出现同一状态或同一功能说明
- 无必要的辅助文案堆叠
- 无意义留白或无意义拥挤
- 仅为了美化而增加用户不需要的视觉层

#### 8.2.5 动效与反馈约束

动效必须服务于状态表达和操作反馈，不得为了“有动效”而动：

- 页面切换、加载、hover、press、toast、弹层等反馈要统一
- 时长、缓动、透明度变化要克制
- 不允许引入与硬件显示冲突的频闪式视觉表达

禁止行为：

- 大面积频繁闪烁
- 无业务意义的装饰性动效
- 同类交互在不同页面使用完全不同的反馈节奏

### 8.3 项目视觉基调补充约束

结合当前项目实际，后续默认遵循以下风格基线：

- 白色为主的极简主义
- 克制、清爽、留白明确
- 强调功能可读性，不靠花哨装饰制造“设计感”
- 图标可有设计感，但不能破坏整体简洁基调

在未得到用户明确确认前，禁止：

- 把页面改成深色主导
- 引入大面积渐变、玻璃拟态、厚重阴影
- 大幅改变成熟页面既有结构
- 为追求“更丰富”而加无关说明、徽章、说明卡片

### 8.4 设计保真工作流

当任务属于“还原设计稿/对齐参考图/替换为新设计”时，必须按以下顺序执行：

1. 先分析参考来源
2. 提取视觉规格
3. 明确哪些内容来自设计稿，哪些是基于现有项目推断
4. 再开始写样式或改结构
5. 交付前做一次保真核对

提取视觉规格时，至少覆盖：

- 颜色系统
- 字体系统
- 间距系统
- 组件规格
- 布局结构
- 阴影/描边/圆角
- 动效与反馈
- 图标风格

### 8.5 保真与回归自检

只要改动了页面视觉或样式，交付前至少完成以下检查：

- 是否与用户给的参考一致，而不是“我觉得更好”
- 是否与项目现有白色极简风格一致
- 是否复用了已有 token、变量或成熟组件样式
- 是否引入了无来源的新颜色、新字号、新圆角体系
- 是否造成了同类组件风格分裂
- 是否增加了无意义文案、重复信息、重复入口
- 是否影响了触控区域、可读性与状态辨识

### 8.6 文档维护规则

当用户后续再补充新的设计规范、Figma 输出规则、图标要求、页面风格禁令时，应优先更新本文件，而不是只停留在对话里。

后续如果出现以下情况，也应同步补充本文件：

- 项目确认新的颜色/字体/间距体系
- 项目确认新的图标导出与使用规范
- 项目确认新的页面层级与信息密度规则
- 项目确认新的动效限制与交互反馈标准
