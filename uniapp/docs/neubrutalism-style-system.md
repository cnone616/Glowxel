# Glowxel 样式系统定板文档

本文件是 `newstyle` 分支样式收口后的执行规范。

目标只有三个：

- 全项目视觉统一
- 只改风格，不乱改布局
- 后续新增页面不再回到散装样式

## 1. 总体原则

后续所有 `uniapp/` 页面都必须遵守：

- 保留原有功能区顺序、操作路径、信息层级
- 编辑页、拼豆模式、画板模式、功能区页面只改风格，不重排布局
- 同一语义的结构只允许保留一套正式样式结构和一套正式类名体系；禁止在不同页面为同一种标题栏、搜索栏、列表入口、工具条各写一套近似实现
- 同一语义组件后续优先通过统一组件或局部 mixin 收口，不允许继续靠页面各自复制粘贴一份样式再微调
- 功能页面的预览区域必须统一留边；黑色预览底板内的像素画布四周可视边距必须使用同一套基准，不允许不同页面一会儿更宽一会儿更窄
- 统一预览留边时，不只看缩放系数，缩放取整规则也必须一致；同样写成 0.96 的页面若一页用整数缩放、一页用浮点缩放，视觉留边仍视为不一致
- 不新增无意义标题、副标题、英文说明
- 标题栏右侧默认不放内容，避免被微信小程序系统区域遮挡
- 不再写轻飘圆角卡片、柔光阴影、玻璃拟态、无来源渐变
- 禁止再通过全局样式、标签级选择器或 `!important` 强行把页面压成统一效果；页面表现必须主要来自页面自身和正式组件自身

## 2. 视觉主系统

当前定板方向：

- 背景主色：`#FFFFFF`
- 卡片背景：`#FFFFFF`
- 主文字：`#000000`
- 次文字：`#444444`
- 主强调色：`#FFD23F`
- 次强调色：`#74B9FF`
- 成功色：`#88D498`
- 危险色：`#FF6B6B`
- 警示色：`#FFA552`

统一结构规则：

- 白色主底
- 粗黑描边
- 硬阴影
- 默认直角
- 标题重，正文稳
- 重点块上色，基础骨架保持白底黑边
- 禁止再把整页、整组表单、整组设置面板铺成淡黄底
- 页面底色统一用白色，只有强调按钮、选中态、状态块才允许上色
- 用户新定板：整体观感要“清新、清晰、干净”，不能把图标底板、入口块和状态块一律做成发灰发淡的浅色片
- 高频页的图标壳必须有明确存在感：要么白底黑边配高对比图标，要么直接使用清晰的强调色底板，不允许再出现“底板像没上色、图标像浮在空气里”的效果
- 帮助弹窗、操作指南、说明卡片里的图标必须和底板形成明确对比；禁止再出现黄底上叠黄图标、结果视觉上像“没有图标”的情况
- 黄色、蓝色、绿色、珊瑚色允许继续使用，但必须更聚焦地服务于主操作、图标底板、状态提示；不允许把整块内容区大面积铺成稀薄彩雾
- “清新”不等于“发白发灰”：次级文字可以弱，但图标、操作入口、筛选选中态、弹窗主按钮必须一眼能识别

## 3. 公共类命名规则

以后新增公共样式时，必须优先使用 `glx-` 前缀，禁止继续发明无前缀的通用类。

已启用的公共类：

- `glx-page-shell`
- `glx-page-shell__fixed`
- `glx-page-shell__stack`
- `glx-page-shell__content`
- `glx-topbar`
- `glx-topbar__title`
- `glx-scroll-region`
- `glx-scroll-stack`
- `glx-preview-panel`
- `glx-preview-panel__info`
- `glx-primary-action`
- `glx-panel-card`
- `glx-panel-head`
- `glx-panel-title`
- `glx-panel-subtitle`
- `glx-tab-panel`
- `glx-choice-chip`
- `glx-choice-chip--compact`
- `glx-tool-tile`
- `glx-action-tile`
- `glx-scene-tile`
- `glx-section-head`
- `glx-section-title`

- `glx-cta-button`
- `glx-list-card`
- `glx-inline-cta`
- `glx-list-meta`

命名约束：

- 页面局部样式允许原类名保留，但跨页面复用的结构必须提成 `glx-` 公共类
- 不允许再用 `card2`、`new-card`、`custom-header` 这种不可维护命名
- 不允许把只适合单页的名字抽成全局公共类

## 4. 重复样式抽取规则

满足以下任一条件，必须提成公共类或公共 token：

- 同一结构在 2 个及以上页面重复出现
- 同一视觉语义在多个页面重复实现
- 同一按钮、标题栏、卡片头、选项块、预览栏重复写一遍
- 同一数据语义的卡片列表，例如 `我的画布 / 我的作品 / 项目列表`，必须共用同一套卡片结构，不允许一边是网格画布卡、一边又手写另一套作品卡

优先抽取的对象：

- 顶部标题栏
- 预览区标题与发送按钮
- 设置卡片
- 卡片头部
- 选项按钮
- 编辑器里的工具块
- 场景卡片
- 设置列表项

禁止事项：

- 为了“局部更快改”复制一整段旧样式到新页面
- 公共类已经能覆盖时，再在页面里重复写一份几乎一样的样式
- 写无作用或已被全局覆盖掉的冗余样式
- 弹窗内部的信息列表要避免“外层弹窗一层边框、内层整卡再一层边框、卡内图标又一层边框”的重复盒子堆叠；优先保留外层主边框和必要的图标底板，减少内容行级整卡描边

## 4.1 形状统一规则

后续不要再混用“圆角卡片”和“直角卡片”：

- 卡片、按钮、入口块、列表项、统计块统一使用直角矩形
- 不再给数量徽标、查看更多按钮、设置入口单独补圆角
- 只有头像、在线状态点、系统原生 `switch` 这类语义上必须为圆形的对象保留圆形
- 如果某个页面已经接入 `glx-` 公共类，页面内不要再额外写一层风格性圆角覆盖

## 4.2 列表入口规则

列表页和设置页右侧入口统一遵守：

- 右箭头颜色跟随主文字，不单独使用灰蓝色
- 文本和箭头必须上下左右居中
- “查看更多 / 查看全部 / 查看记录”统一使用 `glx-inline-cta`
- 列表尾部计数、版本号、箭头组合统一使用 `glx-list-meta`
- 筛选 tabs 的文字禁止拆字换行，像 `已发布 / 审核中 / 草稿` 这类状态词必须单行展示
- tabs 里的数量只做角标或紧凑数字标记，不再把数量做成挤占正文布局的大块结构

## 4.2.1 卡片阴影规则

- 同一组卡片如果使用外部硬阴影，阴影方向和强度必须一致
- 如果栅格边缘导致外部阴影看起来像被裁掉，优先减弱或移除该组卡片阴影，改靠完整边框表达
- 不允许同一行左卡和右卡出现明显不同的阴影视觉

## 4.2.2 图标系统规则

图标系统后续统一按 `IconPark + 项目资产映射` 方向推进，禁止继续混搭多套随手找来的图标。

统一要求：

- `uniapp` 侧图标优先使用已经确认风格的 `IconPark` 资源
- 用户已最新确认：`pixelarticons` 这类单色像素图标库当前不适合作为 Glowxel 项目的最终页面图标语言；可以借鉴其“图标跟随文字色、语义命名、成组管理”的方法，但不作为正式整包接入方向
- 底部导航、设备控制入口、功能入口、设置入口等高频图标必须同一风格族
- 不允许自行重画一个“差不多”的图标去替代成熟图标
- 不允许同一组入口里同时混用线性、填充、拟物、emoji、手绘风
- 图标颜色必须跟随当前组件语义，不能出现图标和文字两套颜色
- 用户已再次强调：同一信息块、同一按钮、同一列表项里的图标与文字默认必须同色；未获明确确认，不允许图标单独上蓝色、橙色、绿色等强调色而文字仍走另一套颜色
- 用户已再次明确：不允许因为怕出错就把全项目图标一律收成黑色；彩色图标、彩色图标底板、彩色文字都可以使用，但必须有明确语义和成组搭配，不能只剩“黑图标 + 黑字 + 白底”这一种偷懒方案
- 图标配色必须按层级设计：主操作、次操作、危险操作、成功状态、提示信息、列表入口、空状态插画可以使用不同彩色方案，但每一类在同组页面内必须保持一致，不能今天蓝图标配黑字、明天黄图标配灰字、后天又全黑
- 用户已最新纠正：`设备控制页` 的 10 个模式入口图标必须一对一区分，不允许多个不同功能继续共用同一个宫格/图片图标冒充占位；提交前必须逐项核对“图标语义”和“肉眼可区分性”
- 用户已最新补充：`画板模式` 入口图标必须明确体现“编辑/绘制”语义，不再使用九宫格这类更像布局或场景列表的图标
- 拼豆辅助页、调色板类页面中的字母分组标题必须统一为白底 + 左侧黄色竖线的样式，不允许单独铺浅蓝渐变底或额外发光效果
- 导航类图标允许多色，但选中后的文字颜色必须和图标主色一致
- 功能配置页里的操作图标以单色高对比为主，不额外叠加花哨彩色底板
- 运行链路只允许引用正式接入目录中的图标资源，像 `download` 这类临时下载目录不得继续作为项目资产来源
- 新增图标前必须先确认是否能复用现有资产映射，避免同义图标越积越多
- 用户已新增确认：正式接入的图标资源必须在图标框内完整、清晰、居中显示；不允许出现“资源存在但实际只在左上角露一个极小点”的不可见渲染效果
- 用户已最新确认：`led_matrix_showcase` 里的宇宙/星云类场景不能只做成“深色背景 + 几颗星点”的普通星空底图；正式场景必须有明确可感知的主体云团、层次分区和视觉焦点，避免平、散、像默认壁纸

## 4.2.3 交互反馈规则

全项目统一禁止“点击摇一下”的交互反馈：

- 禁止 `:active` 下使用 `transform: scale(...)`
- 禁止 `:active` 下使用 `translateX / translateY / translate3d`
- 禁止用按压动画制造按钮、卡片、tabs、返回按钮的位移错觉
- 禁止把图标和文字做不同步的按压变化

统一做法：

- 选中态只通过底色、边框、字重、文字色、图标色表达
- 临时按压态最多只允许颜色轻微变化，不允许位置变化
- 公共组件和公共样式必须先遵守，再覆盖到页面局部
- 用户已再次纠正：同一行按钮允许按语义使用黄底、蓝底、红底等不同底色；真正禁止的是“按钮内部图标和文字不同步”。每个按钮内部都必须保证图标色与文字色完全一致
- 用户已明确当前常用语义：主操作可用黄底黑字黑图标，编辑类可用蓝底白字白图标，删除类可用红底白字白图标；只要按钮内部图标和文字同步，就不视为颜色不统一
- 用户已再次强调：特殊操作按钮不能被“统一按钮”粗暴抹平；像编辑、删除、分享、排序这类动作可以共享结构基线，但必须保留独立语义色和独立状态表达
- 删除/清空这类不可逆确认弹窗里的危险确认按钮默认按“红底白字白图标”收口；不允许再出现红底却配黑字黑图标，或按钮文字和图标各走一套颜色
- 设备控制页这类外层功能入口不做选中态高亮，不允许点击后把文字放大、加粗到换行或改变排版稳定性
- 返回按钮、更多按钮、顶部左右导航按钮不做任何按压反馈，不允许出现缩放、位移、背景跳变或“抖一下”的感觉

## 4.2.4 Cleanroom 结构收口规则

面向后续 `uniapp-cleanroom` 重构，新增以下硬约束：

- 全局样式层只允许保留 design token、基础 reset、safe-area 变量和交互禁令，不承载业务视觉类
- 禁止在全局层定义 `.header`、`.navbar`、`.search-btn`、`.section-title`、`.card-title` 这类业务语义类名
- 禁止用 `page view text button` 之类标签级覆盖把页面压成统一风格
- 标题栏、返回按钮、搜索栏、搜索工具条、自定义 `tabBar` 必须走正式组件，不再允许页面各写一版
- 搜索统一为“标题栏下固定工具层”，不再把搜索入口放到标题栏右侧
- 一级页标题栏允许业务特例，但必须服从同一 safe-area、同一垂直节奏、同一中轴对齐、右侧留空规则
- 所有加载中的首屏态必须使用项目自定义 loading，不再露出系统默认 loading；当前占位方案固定为 `Glowxel Logo + 自定义文案`
- 页面局部样式必须使用组件内部类名或页面自己的 `page-*` 局部类名，不允许继续依赖外层全局类兜底

## 4.3 页面骨架规则

所有带顶部导航的页面，后续必须统一为同一套页面骨架：

- 页面根节点统一使用 `glx-page-shell`
- 顶部栏、搜索栏、tabs 栏、步骤栏、底部操作栏这类不跟着内容滚动的区域统一放在固定层，使用 `glx-page-shell__fixed`
- 主内容滚动区统一使用 `glx-page-shell__content`
- 向导页、多步骤页、双层布局页统一使用 `glx-page-shell__stack` 承接内部滚动区
- 页面只允许“顶部固定，内容区独立滚动”，不允许整页一起滚导致标题栏被顶走
- 搜索栏、分类栏如果位于标题栏下方且需要常驻，也属于固定层，不得混进主滚动区
- 页面内若因弹窗、Toast、连接框等需要临时隐藏原生 canvas，只允许隐藏绘制节点，不允许把整块预览容器直接移出布局；必须保留原占位，避免页面内容发生明显跳动

明确禁令：

- 不允许只把当前正在看的页面改成固定顶部，其他同类页面继续整页滚动
- 不允许页面根节点继续只写 `min-height: 100vh` 却不处理独立滚动关系
- 不允许一部分页面把 tabs、搜索栏放在滚动区外，另一部分页面又放在滚动区内
- 固定搜索栏、筛选栏、底部操作栏的内边距必须与当前页面内容区对齐；不能出现输入框或按钮紧贴分隔线、贴边，或者上下留白一会儿太松一会儿太挤的情况
- 搜索栏这类短工具条默认用紧凑留白，优先保证“输入框清晰、和上下结构对齐”；不允许为了看起来宽松，把工具条本身做成比内容卡片还厚的空白带

## 4.4 新增页面跟随既有体系规则

用户已经明确否定“新增页面另起一套视觉”的做法，后续必须严格执行：

- 新增页面必须优先沿用同组既有页面的骨架、标题栏、分组头、卡片和入口语法，不允许自己重新发明一套页面语言
- 如果页面属于现有业务分支的子页，视觉上必须直接向上对齐该业务的主页面，而不是额外做 `hero`、大圆角、整页渐变、独立徽标区
- `设备参数页` 必须与 `设备控制页` 保持同体系：统一使用同一套 `glx-page-shell`、`glx-topbar`、区块标题、`glx-panel-card`、行内操作卡语法
- `热点配网页 / 配网说明页` 必须保持“简单明了、信息直接”的结构，不允许直接暴露粗糙默认门户样式，也不允许额外做营销化主视觉
- `热点配网页 / 配网说明页` 必须把“连接设备热点 -> 打开 192.168.4.1 -> 选择 2.4GHz WiFi 并保存”放在首屏主信息层，不能把关键提示埋进次级文案
- 样式预览总览页如果用于展示本轮审查结果，页面本身必须直接呈现“审查后的预演效果”和结论，不允许继续停留在历史概念板、旧风格宣言或与当前项目脱节的参考文案上
- 样式预览总览页的配色应优先呈现“高级但有活力”的点缀关系：允许比正式页更活跃，且当用户明确反馈“太淡”时，应优先提升点缀色存在感和张力，但仍不得把黄、蓝、绿、橙做成廉价荧光感或糖果色
- 样式预览总览页必须直接展示关键控件预演，包括滑块、开关、提示条和弹出框；其中滑块预演应体现连续、顺滑的拖动反馈，不能看起来像离散跳格或拖动发顿
- 用户已新增确认：如果样式审查页的目标是“让人直接比对当前和重设计”，默认必须按“单页查看”组织，不允许把多个业务页的类名、控件、案例混成一大张跨页台账
- 用户已新增确认：这类审查页默认固定为“双列对比”结构，左列只放当前真实样式，右列只放重设计后的预演；不使用展开/收起作为主要阅读方式
- 用户已新增确认：右列的“重设计”必须在结构、层级和搭配上肉眼可见地和左列拉开，不能只在原样基础上轻微换色后继续冒充重设计
- 用户已最新确认：样式审查 / 对比页禁止再用大面积淡黄色、米黄色去铺底或制造氛围；黄色只保留给小面积高纯度语义强调，不允许把整页或整列做成“发黄”的观感
- 用户已最新纠正：如果本轮要求“只调整图标”，实现范围就只能落在图标本身，包括图标资源、图标尺寸、图标颜色和图标居中清晰度；不允许顺手把边框、底色、容器结构、说明装饰一起改掉
- 用户已最新补充：如果本轮只点名某个具体页面的颜色收口，例如 `profile` 页，默认只改用户点名的颜色项，例如图标、按钮、菜单区背景；不顺手扩到结构、边框和其他页面
- 用户已最新纠正：列表入口页如果改了图标配色，最终效果里不允许出现“白框里像没图标”的空壳状态；提交前必须先按实际截图自查图标可见性，像 `profile` 这类菜单组背景也不能再铺出一整块脏色底
- `设备参数页` 禁止再出现独立设计语言，例如大面积渐变背景、悬浮主视觉卡、自造信息徽章、和 `control.vue` 不同的按钮/输入框轮廓体系
- 用户已最新纠正：`设备参数页` 里 `旋转 / 色彩 / 夜间开始时间 / 夜间结束时间 / 时间服务器预设` 这类选择框必须只保留外层单一描边，内部值文案不允许再额外套一层小边框；同组控件尺寸必须统一，不允许一项一个大小
- 用户已最新纠正：`设备参数页` 首屏不保留设备摘要卡和独立亮度调节块，页面进入后直接从参数分组开始，不再额外堆一层说明盒子
- 用户已最新收口：`设备控制页` 正式页不再保留独立 `亮度调节`、`样式预览`、`高级/重置网络` 这类辅助区块；页面主结构只保留连接态、模式入口和设备参数入口
- 用户已最新纠正：`设备控制页` 的 `设备参数` 入口卡片不在分组标题右上角额外放说明文案；说明统一放进卡片副标题里，保持标题区干净
- 用户已最新定板：全仓正式页面里的通知、确认弹窗、加载中遮罩统一使用项目自定义组件视觉；不再直接露出系统默认 `toast / modal / loading` 样式
- 当用户已经点名“按已有页面来”时，默认含义就是复用现有页面结构和视觉密度，不允许以“只是样式优化”为理由偏离
- 用户已最新纠正：像 `星球屏保` 这类工具页如果用户明确要求“少解释、少说明”，就不再保留大段提示文案、说明卡、状态口号或“持续播放”这类重复标签；优先把空间让给核心预览和必要控制
- 用户已最新纠正：当页面里存在“随机一个结果”这类动作时，默认用直接按钮表达，不再额外暴露底层编号、内部 seed 值或技术性参数说明给当前用户
- 用户已最新纠正：需要连续细调的速度类参数，不要只给 2-3 个粗粒度 chip 按钮；优先使用和项目现有控件体系一致的步进器或其他紧凑数值控件
- 用户已最新纠正：同一张工具页里的主操作按钮、随机按钮、数值按钮如果处于同一控制层级，视觉尺寸要统一；不允许出现一个按钮明显更宽、更高或更矮的散装状态
- 用户已最新纠正：`星球屏保` 页里的随机按钮文案必须直接表达动作为“随机星球”，不再使用“换一个”这类弱语义口语按钮
- 用户已最新纠正：对当前用户没有判断价值的技术性开关，例如 `像素抖动`，默认直接隐藏，不继续占用页面控制位
- 用户已最新补充：`星球屏保` 的太空背景不必死黑一片，可以适当加入少量低亮度色彩氛围；但背景色层必须克制，不能抢过星球主体
- 用户已最新纠正：`星球屏保` 这类独立功能页的页面骨架必须直接跟随现有 `ambient-editor / game-screensaver / gif-player` 同组页面；不允许单页保留另一套卡片外壳、标题留白或预览栏节奏
- 用户已最新纠正：功能页内的发送按钮、随机按钮等操作控件默认复用现有 `action-btn-sm / glx-primary-action` 尺寸基线；未获明确确认，不允许单页额外写固定宽高覆盖，导致按钮与同组页面或同页其他按钮尺寸不一致

## 5. 编辑页特殊规则

适用页面：

- `clock-editor`
- `ambient-editor`
- `canvas-editor`
- `spirit-screen`
- `tetris-settings`

这些页面后续必须遵守：

- 不改顶部、预览区、内容区的大布局关系
- 不打散原有表单和操作分组
- 优先复用 `glx-panel-card`、`glx-panel-head`、`glx-choice-chip`、`glx-primary-action`
- 局部新增块先看是否属于已有视觉语义，属于就直接复用公共类

### 5.1 功能配置页整组一致规则

`功能配置页` 不允许一页一个样。

凡是属于设备功能配置流的页面，必须按同一套骨架与控件语法实现并整组检查，不允许只修一个页面就提交：

- `clock-editor`
- `ambient-editor`
- `canvas-editor`
- `spirit-screen`
- `tetris-settings`

统一要求：

- 顶部必须使用同一套标题栏骨架：`navbar + nav-left + nav-title + nav-right`
- 标题必须视觉居中，右侧默认留空，不放功能按钮
- 用户已最终确认：`天气看板 / 倒数看板 / 计时看板 / 提醒看板` 当前 uniapp 设计不适合 `64×64` 屏幕，不继续在现有页面上做“补样式式挽救”；直接删掉对应页面文件，并移除正式入口与路由，避免继续以半成品视觉对外展示
- 预览区必须保持同一层级关系：黑底预览画布 + 下方简洁预览栏 + 发送按钮
- 用户已再次明确：`静态时钟 / 动态时钟` 页的首屏预览必须直接可见，不能出现黑底已渲染但没有任何像素内容的状态；预览画布必须接入明确刷新触发，不允许只依赖隐式重绘
- `俄罗斯方块屏保` 设置页恢复旧屏保后，仍必须补齐同组功能页一致的 `64×64` 预览区；禁止只保留参数卡片、不显示预览
- 用户已再次明确：本轮 `俄罗斯方块屏保` 页优先修“功能预览页面设计样式”；预览区必须先完整呈现成熟功能页骨架，再去谈参数区细节，禁止出现“能发但预览没整出来”的半成品观感
- 主题模式的大预览也必须遵守同一预览语法：预览内容在黑底区内居中展示，和其他功能页保持一致的内边距，不允许单独铺满到边缘形成不同留白
- 所有功能页的像素预览画布都必须根据容器尺寸自动计算缩放与偏移，保证在黑底预览区里水平垂直居中，不能贴左上角悬空展示
- 功能页预览画布的内部留白统一控制为紧凑留白，和画板模式同级，不允许某些页面单独留出更厚的黑边
- 所有功能页预览画布必须显示 `64×64` 网格线，用于直观看出预览区域大小与像素对齐；禁止改成默认隐藏
- 网格线视觉必须轻量但清晰，禁止画得过深压住内容，也禁止淡到不可见
- `countdown-board` 预览中的时间文本不再使用额外外框描边，保持小字号像素字与沙漏主体留白协调，不得出现大字压迫感
- `countdown-board` 沙漏外形固定为“上下对立且中心尖端相向（尖对尖）”的双三角，不使用玻璃瓶高光、反射线、盖帽渐层等玻璃质感装饰
- `countdown-board` 的沙粒渲染禁止使用大面积明暗阴影分层去制造“立体玻璃/厚重沙堆”效果；优先使用单一暖黄主色配少量亮点，保持干净像素观感
- 内容区必须使用同一套滚动区、分组卡片、卡片头、选项块、底部 tabs 语法
- 不允许某一页单独改成另一套卡片风格、另一套标题栏、另一套选中态
- 修改其中任意一个功能配置页时，必须同步检查同组其他页面是否已经风格漂移
- 用户已再次明确：`桌面宠物 / 静态时钟 / 动态时钟` 这一组里的时间设置区、字体设置区必须共用同一套成熟语法，不允许只修其中一页
- 用户最新补充：这组页面以及时钟编辑页里复用的文字设置卡，字段顺序也必须统一；当前定板为“字号 / 位置 / 对齐方式 / 颜色”，不要再为了单页需求硬做局部特判
- `countdown-board` 的“快捷时长”选项块必须保证足够触达尺寸与可读性：不允许做成过小按钮，文本不得因控件过窄而显得拥挤
- `countdown-board` 的“快捷时长”必须复用现有功能页选项语法（`option-row + option-btn + glx-feature-option`），禁止再写页面私有按钮外观导致与同组页面边框、阴影、字号语义不一致
- `countdown-board` 页面骨架与内容区背景必须与同组功能页一致，禁止页面局部单独改背景层级或边框粗细，造成“看起来像另一套页面”
- 用户最新定板：`像素场景集` 页不保留分类筛选、来源说明卡和底部 tabs，直接用单页连续布局展示“场景选择 + 参数”
- 用户最新定板：`主题模式` 页的预览区留白要收紧到和同组功能页一致，预览栏不保留副标题，主题卡片也不再显示副标题小字
- 用户已再次纠正：`主题模式` 中间区域不是像素预览链，而是当前所选主题封面图的静态展示；禁止再套用 `PixelCanvas`、网格线或时钟像素预览语义把简单图片展示做复杂
- 用户已再次纠正：`静态时钟 / 动态时钟` 页的本地预览必须直接可见；如果出现黑底但看不到时间像素，视为未修复，不允许以“链路已拆开”作为完成标准
- 用户已再次纠正：`主题模式` 里的主题封面图必须保留明确内边距，按缩略图卡片展示；禁止把图片重新铺满卡片边缘，导致留白被改回去
- 用户最新补充：处理 `画板模式` 边框问题时，默认指内容区外层分组卡片边框，不是撤销/重做、工具按钮、笔触选项、颜色面板输入框这些控件本体边框
- 这组三页里的颜色选择区只保留颜色块、色板和调色面板本体，不允许额外再套一层无语义的大边框、大底板或伪列表项外框
- 这组三页里的字体页必须直接显示可见的字体预览卡，不允许退化成只有文字标签或像原生下拉框一样的空白选择框

明确禁令：

- 不允许只把 `俄罗斯方块时钟`、`桌面宠物`、`像素屏保` 其中一页改对，其他页继续散装
- 不允许某页保留大外边框，而另一页已经改成轻结构
- 不允许某页继续使用旧原生控件视觉，而其他页已经切到新控件
- 不允许某页的 tabs、选项按钮、底部导航继续使用整块黄底选中，而其他页不是
- 对于名称较短、数量较多的预设选择区，优先使用紧凑三列栅格，不要退回两列导致页面被纵向拉长
- 当某个功能页本身分组不多、顺序明确时，不要为了“统一”硬塞底部 tabs；像顺序浏览更自然的配置页，应直接改成单页连续布局
- 计数提示、数量提示这类信息角标属于弱信息，不要再做成浅色漂浮气泡；统一使用白底黑边黑字的紧凑信息徽标
- 主题库里的 `数量 / 当前 / 图片` 这类提示盒也属于同一类弱信息角标：统一使用紧凑直角盒，不再使用半透明漂浮底色
- 在样式调整阶段，设备控制页的功能入口必须支持离线直达，方便逐页查看视觉，不允许未连接时把整组功能入口隐藏掉

### 5.2 功能配置页局部实现细则

后续功能配置页默认按以下细则执行：

- 返回按钮尺寸、描边、居中方式必须一致，不允许有的偏左、有的偏大、有的图标不居中
- 功能页返回按钮统一复用 `navbar + nav-left + nav-title + nav-right`，不允许再混用 `back-btn / header-placeholder / header-content`
- 顶部标题只保留中文标题，不补英文标题、副标题、说明词
- 预览区外层不再额外包一层大边框或大阴影，重点在黑底画布本身
- 预览栏只保留一个核心标题，不再叠加状态 chip、重复标题、解释文案这类冗余信息
- 发送按钮保留强调色，功能页里的可切换按钮选中态统一使用黄底黑字
- 选中态不要额外加左侧色条或装饰条，只通过底色、边框、字重做表达
- 功能页内容区的按钮、选项块、卡片不做按下位移、缩放、抖动动画，只允许静态切换颜色
- 共享样式、公共组件、页面局部都不允许再写点击按压态 `:active` 位移、缩放、抖动或点击触发动画；发现旧规则时必须直接删除原规则，不能靠局部覆盖
- 图标颜色和文字颜色必须同步，不允许图标和文字两套颜色
- 用户已再次明确：所有功能页面里同语义的“选中态”必须使用同一套表现，不允许一部分页面只加粗文字，另一部分页面改成黄底，或者再混入第三套表现
- 后续功能页里的选中态如果属于“可切换按钮 / 选项块 / tab / 底部导航 / 预设卡”这类同语义控件，必须先收敛到统一公共类，再批量替换，不能继续保留页面局部各写各的
- 出现同语义但不同类名的写法时，必须优先收敛为同一套 `glx-` 公共类，不允许继续扩散 `tab-item / category-chip / option-btn / piece-item / tool-option / size-option` 这种平行命名
- 选项块、预设卡内部承载名称的文字也属于同一语义，不允许继续散落 `scene-name / demo-name / preset-option-name / icon-item-name / font-card-name` 这类平行命名；统一收敛到 `glx-feature-option__label` 或 `glx-feature-card-option__label`
- 用户最新定板：所有功能页面的底部 tabs 禁止再使用黄底整块选中态，不要再做一格一格的重边框按钮；当前项只允许通过字重、字色、图标色变深来表达
- 用户最新定板：功能页面内部的弱语义控制开关统一轻量化，像模式切换、对齐切换、颜色预设、小型筛选这类控件不需要重边框盒子；选中态优先只用字重、字色、图标色表达，不再默认给每个小控件套硬边框和黄底
- 当用户要求“先看所有功能页面再动手”时，必须先按功能页全集逐页审查同类控件，再统一修改；不允许只改用户截图页后让其他页面继续保留旧语法
- 用户最新定板：功能页面默认不要保留底部 tabs；只有分组切换确实有信息价值、明显优于连续布局时才允许存在。像“预设 / 参数 / 字体”这类顺序浏览即可完成的内容，优先直接改成单页连续布局，不要为了统一硬保留 tabs
- 用户已经明确分好组、并且页面信息架构已经稳定的功能页，禁止擅自删除其底部 tabs 或打散原分组；尤其 `桌面宠物 / 静态时钟 / 动态时钟` 这组，必须保留用户已经确认过的分组结构，只允许调整选中态表现，不允许把已确认的信息架构误改成连续布局
- 当用户指出“这个页面我已经分好了”时，后续必须先恢复该页原有分组，再继续讨论哪些其他页面属于无效 tabs；不允许把“删掉没用 tabs”的规则扩大解释到所有功能页
- 用户最新纠正：本轮要去掉的是内容分组外层那层“大盒子边框”，不是内部按钮、选项按钮、预设按钮、表情按钮本身的边框；按钮边框属于有效信息结构，禁止误删
- 如果用户说“这个边框不要了”，而页面里同时存在外层分组边框和内层按钮边框，必须先确认用户指的是哪一层；未确认时默认只动外层分组盒，不动按钮本体
- `桌面宠物 / 静态时钟 / 动态时钟` 的底部 tabs 是用户已经确认过的分组结构，必须保留；本轮只允许调整 tabs 的选中表达，不允许再删除

## 10. 样式隔离重构规则

用户已最新确认：如果要新建 `uniapp` 重构工程，目标不是继续叠公共样式，而是先把当前项目的样式结构和污染链路理清，再按“页面自管”重建。

执行规则：

- 新工程默认不继承旧 `App.vue` 的全局样式导入链；禁止把 `theme.css / neubrutalism-global.css / glx-style-system.css / iconfont.css` 原样整包接入新工程
- 新工程里的页面样式优先级最高；页面内部视觉应先由页面自身 `<style scoped>` 或页面私有组件样式负责，不允许被工程外层共享样式反向改写
- 若某条规则只服务单页，就只写在该页；不允许为了少写几行样式，把单页结论抽成全局类再去污染其他页
- 新工程默认禁止新增“共享按钮皮肤”一把套全站；按钮必须按页面和作用分别设计，不能继续靠同一个公共类把主操作、编辑、删除、跳转按钮全部压成一套外观
- 同一页面内的按钮也必须按作用区分，不允许只改文案不改视觉语义
- 非必要不允许使用 `!important`、多层祖先选择器、标签全局覆盖、批量类名并写、或一组属性强行压住页面表现
- 非必要不允许同时使用多组互相兜底的尺寸、颜色、边框、阴影去“强制定住”样式；优先保留最少、最直接、最能说明语义的属性
- 新工程若确需共享层，只允许保留“不会改视觉结果”的最小基础层，例如平台必要变量或极少量页面容器约束；不允许共享层直接定义按钮、卡片、导航、图标壳等正式视觉
- 旧项目样式结构必须先做成清晰审计，至少说明：入口文件、全局导入链、变量层、公共样式层、页面局部层、组件层以及当前已知污染点；未审清前，不允许宣称“可以直接迁”

禁止事项：

- 禁止在新工程继续使用跨页面命名的视觉类去统一按钮皮肤
- 禁止通过全局 `page / view / button / text` 选择器直接规定业务页视觉
- 禁止通过外层公共样式修改页面内部图标、按钮、菜单、卡片的最终颜色和边框
- 禁止为了压过旧样式，在页面内继续叠更多 `!important` 或更长选择器链
