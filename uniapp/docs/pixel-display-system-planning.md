# Glowxel 64x64 像素展示系统规划文档

## 1. 文档目的

本文档用于沉淀 Glowxel 当前 `64x64 像素展示` 方向的统一规划，明确以下内容：

1. 当前已经具备的能力
2. 当前架构的优势与限制
3. 是否适合借鉴或移植 `Clockwise` 一类项目
4. 后续功能应该如何分层
5. 哪些功能应优先推进
6. 哪些功能暂不建议投入

说明：

1. 本文档聚焦 `像素展示` 主线，不扩展到通知、智能家居联动、消息卡片等非核心能力。
2. 本文档是规划文档，不在本文中定义新的协议字段。
3. 若后续涉及接口参数变更或板载协议调整，必须另行输出参数映射表后再实施编码。

## 2. 当前目标重申

Glowxel 当前像素展示方向，应明确为以下主线：

1. `uniapp` 页面负责上传、编辑、预览、保存、发送
2. 页面通过 `WebSocket` 控制板载程序
3. 板载程序负责模式切换、像素渲染、动画播放
4. 核心目标是 `64x64 像素展示效果`
5. 当前阶段不做通知消息、智能家居联动、状态卡片体系

换句话说，Glowxel 不是要做一个大而全的信息屏生态，而是要做一个：

1. 以 `像素展示` 为主
2. 以 `时钟 / 主题 / 素材 / 算法效果` 为核心
3. 通过 `WS` 驱动板载渲染
4. 强调 64x64 小屏观感的像素产品

## 3. 当前已具备的能力盘点

### 3.1 前端设备控制链路

当前 `uniapp` 已具备以下能力：

1. 保存设备 IP、维护连接状态
2. 通过 WebSocket 与板载程序直连
3. 切换设备模式
4. 发送完整图像、稀疏像素、局部更新
5. 获取设备状态并缓存当前业务模式

现状判断：

1. 设备连接链路已经成型，不需要推倒重做。
2. 后续重点不在通信层，而在展示层和主题层。

### 3.2 时钟编辑能力

当前时钟编辑链路已经具备：

1. 静态时钟和动态时钟两个独立模式
2. 时间、日期、星期的显隐、位置、颜色、对齐设置
3. 字体、秒显示、12/24 小时制设置
4. 图片上传与背景叠加
5. GIF 上传、解析、预览、播放、发送
6. 本地保存配置与再次恢复
7. 64x64 预览

现状判断：

1. 时钟编辑器已经是可用链路，不应重做。
2. 后续应在此基础上增加 `主题能力`，而不是改成另一套产品形态。

### 3.3 画板模式

当前画板模式已经具备：

1. 页面标题
2. 64x64 网格预览
3. 黑色背景预览
4. 拖动、绘画、擦除
5. 画笔颜色选择
6. 笔触大小
7. 缩放、适配、撤销、重做、清空
8. 本地保存与发送到设备

现状判断：

1. 画板模式已经是一条完整的纯绘制能力链路。
2. 画板模式应继续保持 `纯像素绘制工具` 定位。
3. 不应再把时间、日期、素材入库等内容重新混入该页面。

### 3.4 算法效果能力

当前效果编辑页已经具备或正在承接：

1. 文本展示
2. 呼吸灯
3. 律动
4. 独立的效果预览
5. 保存和发送

现状判断：

1. 这部分适合作为 `算法类效果` 的承载页。
2. 不适合继续与素材导入混用。

### 3.5 素材入库工具

当前已在 `esp32-firmware` 侧新增独立素材入库页，具备：

1. 图片 / GIF 上传
2. 64x64 网格预览
3. 黑色背景预览
4. 宽高与 XY 调整
5. GIF 播放控制
6. 输入素材名称
7. 保存到本地素材文件
8. 读取已保存素材列表
9. 删除素材
10. 点击素材后二次预览

当前本地存储方案：

1. 使用独立目录 `animation-presets/`
2. 一个素材对应一个文件
3. 保存时检索文件名避免重复

现状判断：

1. 素材工具方向正确，且应继续保留为独立工具页。
2. 它的职责是 `素材制作与入库`，而不是终端用户的主功能页面。

## 4. 当前架构的关键判断

## 4.1 当前架构不是问题

Glowxel 当前主链路是：

1. 页面制作配置
2. 页面预览 64x64 效果
3. 通过 WS 将结果发送给板载
4. 板载按当前模式进行渲染或播放

这个架构本身没有问题，反而很适合继续演进。

当前不需要重做的部分：

1. `uniapp` 页面总体结构
2. WebSocket 控制链路
3. 画板模式编辑逻辑
4. 静态时钟 / 动态时钟的区分方式
5. 素材入库页的独立定位

## 4.2 当前真正缺少的是“展示分层”

当前系统最需要补上的，不是页面数量，而是产品分层。

后续像素展示系统应明确拆成以下三类核心内容：

1. `主题`
2. `素材`
3. `算法效果`

如果这三类不分清，后面会持续出现以下问题：

1. 页面职责混乱
2. 板载资源越来越重
3. 很多内容看起来都能做，但实际不好维护
4. 新增一个效果时，不知道应该放在哪一层

## 5. 对外部项目的借鉴结论

## 5.1 对 Clockwise 的判断

结论：

1. `Clockwise` 非常值得借鉴
2. 但不建议完整移植

原因：

1. `Clockwise` 的强项是 `多主题 clockface 表现`
2. 它适合借鉴的是 `整屏布局、主题感、clockface 组织方式`
3. 它不适合直接作为 Glowxel 当前固件体系的整包替换方案
4. Glowxel 当前板载程序更偏 `模式播放器`，而非 `重资源主题运行时`
5. 如果直接完整搬运，容易造成资源体积增大、维护复杂度提升、板载负担过重

正确借法：

1. 借主题理念
2. 借整屏视觉表现
3. 借几种成熟布局
4. 用 Glowxel 现有 `WS + 模式切换 + 播放器` 架构重新实现轻量版主题系统

## 5.2 对其他项目的借鉴结论

### 5.2.1 `HUB75 Pixel Art Display`

适合借：

1. GIF / 图片上传流程
2. 素材管理思路
3. 素材预览和列表控制

### 5.2.2 `ESP-Smart-Display`

适合借：

1. 编辑模式页面组织
2. 组件式配置思路
3. 后台控制面板结构

### 5.2.3 `WLED`

适合借：

1. 像素素材导入思路
2. 预设和播放列表思路
3. 矩阵效果组织思路

不适合借：

1. 整体产品形态
2. 灯效控制台式 UI

## 6. 核心展示分层规划

## 6.1 主题层

### 6.1.1 定义

主题层不是资源包，而是 `布局模板 + 样式规则 + 内容引用关系`。

### 6.1.2 主题应负责什么

主题应负责：

1. 时间显示区域布局
2. 日期和星期是否显示
3. 字体风格选择
4. 主视觉区域布局
5. 是否使用背景素材动画
6. 是否叠加算法效果
7. 是否有边框、角标、装饰层

### 6.1.3 主题不应负责什么

主题不应负责：

1. 直接携带大量位图资源
2. 携带完整重动画包
3. 内置大量不可复用素材

### 6.1.4 规划结论

1. Glowxel 必须做主题层
2. 主题层是实现“像 Clockwise 一样多主题”的关键
3. 主题必须做成轻量模板，而不是重资源包

## 6.2 素材层

### 6.2.1 定义

素材层用于承载 `图片 / GIF 导入后的像素帧数据`。

### 6.2.2 素材层适合做什么

1. 导入外部 GIF 或图片
2. 调整尺寸和位置
3. 转换为 64x64 像素帧数据
4. 保存为素材文件
5. 提供预览与删除
6. 供主题或动画模式引用

### 6.2.3 素材层适合承接哪些内容

1. 复杂烟花
2. 角色动作
3. 节日小场景
4. Logo 动画
5. 外部寻找的优质像素 GIF
6. 算法难以做好看的复杂动态效果

### 6.2.4 规划结论

1. 当前素材入库页的方向正确
2. 后续重点不是再做更多页面，而是把素材真正接入主题调用链

## 6.3 算法效果层

### 6.3.1 定义

算法效果层用于承载 `由板载程序实时生成` 的动态表现。

### 6.3.2 适合算法化的内容

1. 呼吸
2. 律动
3. 流星
4. 波纹
5. 雨滴
6. 雪花
7. 扫描线
8. 星闪
9. 火焰
10. 简单烟花

### 6.3.3 不适合优先算法化的内容

1. 复杂烟花演出
2. 角色动作帧动画
3. 复杂场景循环
4. 对造型要求很高的节日动画

### 6.3.4 规划结论

1. 算法效果应继续放在板载端
2. 算法效果负责“省空间、可调参数、稳定播放”
3. 不应强行用算法替代所有素材动画

## 7. 板载程序角色定位

Glowxel 板载程序应继续保持 `轻量渲染引擎` 定位。

板载程序建议长期固定承担以下职责：

1. 模式切换
2. 时钟渲染
3. 画板渲染
4. 动画播放
5. 算法效果执行
6. 主题应用
7. 少量本地文件读取
8. 开机恢复最近状态

板载程序不建议承担：

1. 大量重主题资源管理
2. 复杂素材库检索系统
3. 大量在线业务逻辑
4. 很重的页面端逻辑

## 8. 当前已发现的架构限制

## 8.1 动画仍有帧数限制

当前动态时钟链路中，动画仍然存在明显限制：

1. 前端 GIF 转板载数据时仍带有 `20 帧` 限制
2. 板载加载动画时也会限制为 `20 帧`

这意味着：

1. 当前 Glowxel 动画播放能力还不适合直接承载大量长素材
2. 后续若要提升素材表达力，必须专项处理动画限制问题

## 8.2 板载动画存储更像“单动画文件”

当前板载程序更接近：

1. 接收一次动画
2. 保存为本地动画文件
3. 作为当前动画播放

而不是：

1. 拥有成熟多素材库
2. 可随时索引多份动画文件
3. 拥有完整主题包管理能力

这意味着：

1. 短期内最适合做“轻主题 + 少量素材引用”
2. 不适合立刻做庞大的板载素材库系统

## 9. 功能规划清单

本节按 `保留强化 / 必做新增 / 后置 / 暂不做` 分类。

## 9.1 保留并强化

### 9.1.1 静态时钟

结论：保留并强化

强化方向：

1. 增加主题预设
2. 增加布局预设
3. 增加快捷样式
4. 保持当前链路不被破坏

### 9.1.2 动态时钟

结论：保留并强化

强化方向：

1. 继续承接 GIF / 动图背景能力
2. 继续支持时间叠加
3. 后续接入素材引用能力

### 9.1.3 画板模式

结论：保留

强化方向：

1. 继续保持独立
2. 不混入时间参数
3. 强化纯绘制体验即可

### 9.1.4 文本展示 / 呼吸 / 律动

结论：保留

强化方向：

1. 作为算法效果体系的基础能力
2. 后续考虑增加更多算法类展示

### 9.1.5 素材入库工具

结论：保留并继续完善

强化方向：

1. 提高与板载播放链路的衔接
2. 提高素材筛选效率
3. 保持独立工具属性

## 9.2 V1 必做新增

### 9.2.1 轻量主题系统

优先级：最高

必须实现的原因：

1. 这是 Glowxel 走向多主题展示的核心
2. 这是借鉴 `Clockwise` 的正确落点
3. 这能避免“每次靠手调参数切出风格”

### 9.2.2 主题选择入口

优先级：高

建议：

1. 第一阶段直接接入时钟编辑器
2. 不急着独立做一个大型主题管理后台

### 9.2.3 第一批主题预设

优先级：高

建议首批控制在 `4 到 6 个`，例如：

1. 极简数字时钟
2. 边框时钟
3. 中央主视觉 + 角标时间
4. 顶部时间 + 底部动画
5. 节日海报风
6. 像素游戏风

### 9.2.4 素材被主题引用

优先级：高

说明：

1. 素材工具已经能保存素材
2. 下一步必须让主题或动态时钟真正引用这些素材

### 9.2.5 主题与算法效果组合

优先级：中高

建议：

1. 一期先支持 `一个主题最多叠一个算法效果`
2. 避免一开始允许任意叠加导致管理失控

## 9.3 V1 应优化但不抢主线

### 9.3.1 动画帧数限制专项处理

优先级：中高

说明：

1. 若不处理，该问题会限制很多素材的表现力
2. 但它属于专项技术问题，可在主题系统之后推进

### 9.3.2 素材与板载测试链路打通

优先级：中

说明：

1. 后续应支持素材入库后直接投到板载测试
2. 避免在多个页面间频繁切换

### 9.3.3 主题预览一致性

优先级：中

说明：

1. 页面预览应尽量接近板载真实显示
2. 否则主题筛选成本会很高

## 9.4 V2 后置能力

### 9.4.1 完整主题编辑器

说明：

1. 类似 `Canvas JSON` 或可视化主题编辑器的能力，后续可以做
2. 但当前阶段投入过大，不宜抢前期主线

### 9.4.2 多主题资源包体系

说明：

1. 当前不适合做完整资源包系统
2. 待主题模板体系稳定后再考虑

### 9.4.3 板载素材库系统

说明：

1. 当前板载更适合少量本地资源
2. 不建议立即做复杂多文件检索和调度系统

### 9.4.4 在线下载主题或素材

说明：

1. 当前阶段不需要
2. 后续若产品形态成熟，再决定是否引入

## 9.5 当前阶段明确不做

1. 通知消息卡片体系
2. 智能家居联动
3. 大型状态看板
4. 大而全信息屏生态
5. 完整搬运外部项目的全部主题与资源
6. 在板载程序中塞入大量重资源包

## 10. V1 推荐产品结构

Glowxel V1 像素展示系统建议由以下模块组成：

### 10.1 时钟编辑页

职责：

1. 静态时钟配置
2. 动态时钟配置
3. 主题选择
4. 主题预览
5. 图片 / GIF 背景叠加

### 10.2 画板模式页

职责：

1. 纯像素绘制
2. 本地保存
3. 发布到设备

### 10.3 效果编辑页

职责：

1. 算法效果调整
2. 效果预览
3. 发送到设备

### 10.4 素材入库工具页

职责：

1. 导入 GIF / 图片
2. 调整为 64x64 像素素材
3. 保存成本地文件
4. 素材列表展示
5. 素材预览

### 10.5 板载程序

职责：

1. 应用当前主题
2. 播放当前素材
3. 执行算法效果
4. 响应页面通过 WS 下发的控制命令

## 11. V1 实施顺序建议

建议按照以下顺序推进：

### 阶段 1：先定主题边界

目标：

1. 明确主题只管什么
2. 明确主题不管什么
3. 明确主题和素材、算法的边界

### 阶段 2：建立第一版轻量主题系统

目标：

1. 在现有时钟编辑器上增加主题选择能力
2. 做出第一批主题预设
3. 不改变现有时钟主链路

### 阶段 3：把素材层真正接入主题层

目标：

1. 让已保存素材能被主题或动态时钟调用
2. 让素材真正成为产品资产，而不仅是工具页数据

### 阶段 4：优化动画限制

目标：

1. 评估并提升长素材的可播放能力
2. 处理帧数限制、存储限制、内存限制

### 阶段 5：再考虑高级主题编辑

目标：

1. 进一步提高主题自定义能力
2. 在轻量体系稳定后再考虑更复杂的主题编辑器

## 12. 规划结论

最终结论如下：

1. Glowxel 当前 `uniapp + WS + 板载模式渲染` 的总体方向是正确的。
2. 当前上传、预览、画布配置、素材入库这些页面能力并不是问题，不需要推倒重做。
3. 当前最缺的是 `主题层`，而不是更多零散页面。
4. `Clockwise` 值得借鉴，但应借鉴其 `主题感和整屏布局`，不建议完整移植。
5. Glowxel 应建立 `主题 + 素材 + 算法效果` 三层结构。
6. 板载程序应继续保持轻量，不应变成重资源主题运行时。
7. V1 的最优方向是先做 `轻量主题系统`，再打通素材引用，最后再处理长动画和高级主题编辑。
