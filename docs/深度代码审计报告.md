## 一、🔴 必须修复的严重问题

### BUG-S1: ESP32 启动时 AnimationManager 强制覆盖模式 [逻辑错误]

**文件**: `esp32-firmware/src/animation_manager.cpp:19-27` + `main.cpp:29-31`

**问题**:
```
main.cpp 启动顺序:
  1. ConfigManager::init()  → loadClockConfig() → 从 Preferences 恢复 dispMode=CANVAS
  2. AnimationManager::init() → loadAnimation() → 强制 currentMode = MODE_ANIMATION ⬅️ 覆盖了
```
用户在静态时钟模式下重启板子 → 如果 LittleFS 里有旧动画文件 → 会被强制切到动画模式，违反用户预期。

**修复**:
```cpp
// animation_manager.cpp init()
if (loadAnimation() && currentGIF != nullptr) {
  Serial.printf("已恢复保存的动画: %d 帧\n", currentGIF->frameCount);
  // 只在模式确实是 ANIMATION 时才自动播放（模式由 ConfigManager 恢复）
  if (DisplayManager::currentMode == MODE_ANIMATION) {
    currentGIF->isPlaying = true;
    currentGIF->currentFrame = 0;
    currentGIF->lastFrameTime = millis();
    renderGIFFrame(0);
  }
  // 不要覆盖 currentMode！
}
```

### BUG-S2: loadAnimation() 用 new 分配，freeGIFAnimation() 用 free 释放 [内存UB]

**文件**: `animation_manager.cpp:533-535` vs `animation_manager.cpp:85-87`

```cpp
// loadAnimation() 里:
currentGIF = new GIFAnimation();             // ← new
currentGIF->frames = new AnimationFrame[frameCount]; // ← new[]

// freeGIFAnimation() 里:
free(currentGIF->frames);  // ← free() 释放 new[] 分配的内存 → 未定义行为
free(currentGIF);           // ← free() 释放 new 分配的内存 → 未定义行为
```
C++ 标准要求 `new` 配 `delete`，`new[]` 配 `delete[]`，`malloc` 配 `free`。混用是**未定义行为**，在 ESP32 上可能导致堆损坏并随机崩溃。

**修复**: 统一用 `malloc`/`calloc`/`free`（与其余代码风格一致）:
```cpp
currentGIF = (GIFAnimation*)malloc(sizeof(GIFAnimation));
memset(currentGIF, 0, sizeof(GIFAnimation));
currentGIF->frames = (AnimationFrame*)calloc(frameCount, sizeof(AnimationFrame));
```

### BUG-S3: WebSocket 断开后画布数据丢失 [用户数据丢失]

**文件**: `websocket_handler.cpp:35-53`

```cpp
else if (type == WS_EVT_DISCONNECT) {
  if (AnimationManager::currentGIF != nullptr && ...) {
    // 有动画，保持动画模式
  } else {
    DisplayManager::currentMode = MODE_CANVAS;
    DisplayManager::canvasInitialized = false;  // ← 标记画布为未初始化
    // 释放 blackPixels...
    DisplayManager::displayClock();  // ← 直接显示时钟，覆盖画布
  }
}
```
**场景**: 用户在画板编辑器中画了一幅像素画 → 手机锁屏/切后台 → WebSocket 断开 → 板子上的画作消失，变成时钟。用户解锁手机重连后，画布已被清空（`canvasInitialized = false`）。

**修复**: 断开连接时不应清空画布，因为画布数据已存在 `canvasBuffer` 中：
```cpp
else if (type == WS_EVT_DISCONNECT) {
  DisplayManager::clientConnected = false;
  // 不要改变模式或清除画布，保持当前显示状态
  // 用户重连后可以继续编辑
}
```

### BUG-S4: showSparseImage/sendPartialUpdate 偏移量硬编码为 52x52 [逻辑错误]

**文件**: `webSocket.js:320-321` + `webSocket.js:370-371`


### BUG-S7: user.js 路由顺序导致 /settings 永远不可达 [后端逻辑]

**文件**: `server/src/routes/user.js:132` vs `user.js:158`

### BUG-S9: API 响应格式不一致，前端解析错误 [前后端联调]

**文件**: `server/src/routes/user.js:82` vs `uniapp/api/request.js:48-52` vs `uniapp/store/user.js:94`

```javascript
// 后端返回:

//                    ^^^^  data.user
```

### BUG-M4: ConnectModal 组件直接操作父组件状态 [反模式]

**文件**: `uniapp/components/ConnectModal.vue:145`


### BUG-M6: 后端 rateLimitStore 内存泄漏风险

**文件**: `server/src/middleware/auth.js:60-92`

限流用的是个普通 JS 对象 `rateLimitStore = {}`，虽然有 5 分钟清理，但如果遭受大量不同 IP 的请求（比如 DDoS 或爬虫），对象会膨胀到吃满内存后 Node 进程崩溃。

### BUG-M7: displayClock() 每分钟全量重绘背景图 [性能]

**文件**: `display_manager.cpp:85-94`

```cpp
// 每次 displayClock() 都会重绘全部背景像素
for (int i = 0; i < ConfigManager::imagePixelCount; i++) {
  dma_display->drawPixelRGB888(pixel.x, pixel.y, pixel.r, pixel.g, pixel.b);
}
```
如果背景有 4096 (64×64) 个像素，每分钟都要全部重绘一遍，加上 yield() 调用，整个过程耗时明显。背景图在两次时钟刷新之间不会变化，每次都重绘是浪费。

**修复**: 只在背景图变化时绘制，时钟刷新时只擦除+重绘时钟文字区域的背景像素。

### BUG-M8: 后端 SHA256 密码哈希不加盐或盐太弱 [安全]

**文件**: `server/src/routes/user.js:10`

```javascript
const hashPassword = (pwd) => crypto.createHash('sha256')
  .update(pwd + (process.env.PWD_SALT || 'glowxel'))
  .digest('hex')
```

1. 默认盐 `'glowxel'` 是硬编码的公开值，等于没有盐
2. SHA256 不是密码哈希算法（太快，GPU 可以每秒算几十亿次）
3. 没有 per-user salt，相同密码哈希值相同

**修复**: 使用 `bcrypt` 或 `argon2`。

## 四、架构层面问题

### ARCH-1: UniApp 前端与后端响应格式不统一

后端使用 `{ code: 0, data: {...} }`，前端 `request.js` 转换为 `{ success: true, data: ... }`。但转换逻辑 `data: res.data.data || res.data` 有歧义：当后端返回 `{ code: 0, data: null }` 时，前端 fallback 到整个 `res.data`，结构完全错误。

**建议**: 统一规范，前端严格用 `res.data.data`，不做 fallback。

### ARCH-2: ESP32 WebSocket 不支持断线重连 / 心跳

UniApp 端的 `webSocket.js` 没有实现断线自动重连、心跳检测、指数退避。手机WiFi切换/板子重启后不会自动重连，用户必须手动重新输入IP。

### ARCH-3: 像素数据存储在 NVS Preferences（大小受限）

ESP32 的 Preferences 底层是 NVS，默认分区只有 20KB。64×64 像素 × 5 字节 = 20KB，刚好打满。加上其他配置数据很容易超限。**建议**改用 LittleFS 文件存储。

### ARCH-4: 发布入口仍需登录但无优雅提示

创建项目已不再要求登录，但用户创建完项目后点发布时仍需登录，应在发布入口处提示而非拦截创建。

### ARCH-5: Website 官网编辑器无 WebSocket 能力

`website/src/views` 有编辑器页面，但无法与板子通信，展示与实际功能断层。
