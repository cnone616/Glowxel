# Glowxel 像素艺术平台 - 项目架构与接口文档

## 一、项目概述

Glowxel 是一个像素艺术创作 + LED 矩阵屏控制的微信小程序，包含：
- 像素画创作编辑器（支持多尺寸、调色板、分板管理）
- LED 矩阵屏实时控制（WebSocket 通讯 ESP32）
- 时钟模式（静态/动态 GIF 时钟）
- 社区分享（作品展示、评论、点赞、收藏、关注）
- 挑战活动系统
- 模板库
- 云同步

## 四、数据存储现状与问题

### 4.1 当前存储方式

| 数据 | 存储位置 | 问题 |
|------|----------|------|
| 用户信息 | `uni.storage` (isLoggedIn, userInfo, currentUser, loginTime) | 无真实鉴权，Mock数据 |
| 项目列表 | `uni.storage` (projects) | 本地存储，容量有限(10MB) |
| 像素数据 | `uni.storage` (pixels-{id}) | 大项目可能超出存储限制 |
| 看板进度 | `uni.storage` (board-progress-{pid}-{bid}) | key分散，难以批量管理 |
| 看板缩略图 | `uni.storage` (board-thumbnail-{pid}-{bid}) | base64占用大量空间 |
| 设备列表 | `uni.storage` (devices) | 本地管理，无云端同步 |
| 设备IP | `uni.storage` (device_ip) | 仅缓存最后连接的IP |
| 设备模式 | `uni.storage` (device_mode) | 前端缓存 |
| 同步设置 | `uni.storage` (sync-settings) | 云同步功能全部是模拟的 |
| 社区数据 | Mock静态数据 (data/mock/) | 无后端，全部硬编码 |
| 云开发 | `utils/cloud.js` + `api/project.js` | 已写好微信云开发接口但未使用 |

### 4.2 核心问题

1. 社区功能全部依赖 Mock 数据，无法真实交互
2. 项目数据全部存本地 storage，换设备数据丢失
3. 云同步页面是纯 UI 壳，所有操作都是 setTimeout 模拟
4. 用户登录是假登录，openid 写死 `mock_user_999`
5. 发布作品只是修改本地 project.status，没有真正上传到服务器
6. 像素数据用 Map 序列化存储，大项目(64x64+)可能超出 storage 限制

| ping | `{ cmd: "ping" }` | 心跳检测 |
| status | `{ cmd: "status" }` | 获取设备状态，返回模式/亮度等 |
| set_mode | `{ cmd: "set_mode", mode: "clock\|animation\|canvas" }` | 切换模式 |
| brightness | `{ cmd: "brightness", value: 0-255 }` | 设置亮度 |
| clear | `{ cmd: "clear" }` | 清屏 |
| pixel | `{ cmd: "pixel", x, y, r, g, b }` | 设置单个像素 |
| text | `{ cmd: "text", text, x, y }` | 显示文字 |
| set_clock_config | `{ cmd: "set_clock_config", config: {...} }` | 时钟配置 |

### 7.2 二进制数据

像素批量传输使用二进制格式，每像素 5 字节：`[x, y, r, g, b]`，每批 64 像素，批间隔 200ms。

### 7.3 GIF 动画传输

通过 JSON 分帧发送：`{ cmd: "animation_frame", index, delay, width, height, pixels: [r,g,b,...] }`

## 十、新增功能需求分析

### 功能一：导入像素图纸（带坐标轴和颜色标注的图纸照片）

#### 10.1 需求描述

用户拍照或从相册选择一张「像素图纸」（纸质打印的图纸，上面有坐标轴网格和颜色编号标注），系统自动识别并还原为可编辑的像素项目。

#### 10.2 核心难点

1. 图纸拍照存在透视变形、光照不均、阴影、纸张褶皱
2. 需要精确识别网格线和坐标轴，定位每个格子
3. 需要识别格子内的颜色编号文字（如 Artkal 色号 C01、S12 等）
4. 颜色编号必须与 Artkal 调色板精确对应，不能出错
5. 不同图纸格式可能不统一（我们自己导出的 vs 第三方图纸）

#### 10.3 技术方案

```
拍照/选图 → 图像预处理 → 网格检测 → 格子分割 → 颜色/编号识别 → 生成像素数据
```

方案 A：纯前端处理（适合我们自己导出的标准图纸）
- 利用已知的图纸格式（exportCanvas.js 导出的格式是固定的）
- 图纸有固定的标题区、坐标轴区、网格区、图例区
- 通过 Canvas 读取像素，按已知网格尺寸切割
- 每个格子取中心区域颜色，与 Artkal 调色板做最近色匹配
- 如果格子内有文字编号，用颜色编号直接查表

方案 B：后端 OCR + CV 处理（适合任意图纸）
- 前端上传图片到后端
- 后端用 OpenCV 做透视矫正、网格检测
- 用 OCR（Tesseract/PaddleOCR）识别格子内的颜色编号
- 返回结构化的像素数据给前端

建议：先做方案 A（识别自己导出的图纸），后续扩展方案 B。

#### 10.4 涉及的现有代码

| 文件 | 关系 |
|------|------|
| `pages/create/create.vue` | 新建项目入口，已有「导入图片」功能，需新增「导入图纸」入口 |
| `utils/imageImport.js` | 已有图片导入逻辑，需扩展图纸识别 |
| `utils/exportCanvas.js` | 导出图纸的格式定义（反向解析的依据） |
| `data/artkal-colors-full.js` | Artkal 色号对照表，识别后查表用 |
| `store/project.js` | addProject + saveProjectPixels 创建项目 |

#### 10.5 新增代码

| 新增 | 说明 |
|------|------|
| `utils/chartImport.js` | 图纸识别核心逻辑：网格检测、格子分割、颜色匹配 |
| `pages/create/create.vue` 新增模式 | mode='chart' 图纸导入模式，拍照→预览→确认→创建项目 |
| 后端接口（可选） | `POST /api/chart/recognize` 上传图纸图片，返回识别结果 |

#### 10.6 数据流

```
用户拍照/选图
  ↓
Canvas 读取图片像素数据
  ↓
检测图纸边界（标题栏、坐标轴位置）
  ↓
计算网格参数（cellSize、起始偏移、行列数）
  ↓
逐格提取：取格子中心区域主色
  ↓
与 Artkal 调色板做最近色匹配（欧氏距离）
  或 OCR 识别格子内文字编号 → 直接查表
  ↓
生成 Map<"x,y", colorCode> 像素数据
  ↓
调用 projectStore.addProject() 创建项目
调用 projectStore.saveProjectPixels() 保存像素
  ↓
跳转到编辑器，用户可校验和修正
```

### 功能二：作品发布生成带水印图纸 + 收藏作品只读模式

#### 10.7 需求描述

分两部分：

A. 用户发布作品后，后台自动生成一张「带水印的图纸」：
- 基于像素数据生成标准图纸（坐标轴 + 颜色编号 + 网格）
- 叠加 Logo 水印 + 作者用户名水印
- 防止他人直接截图使用
- 这张图纸作为作品的展示图/封面

B. 其他用户可以「收藏」或「喜欢」别人的作品，添加到自己的项目列表：
- 收藏后在「我的项目」中显示，但标记为「只读收藏」类型
- 只能查看和拼豆（辅助视图），不能编辑像素
- 不能发布、不能导出图纸、不能下载、不能分享像素数据
- 防止用户白嫖别人的创作

#### 10.8 功能 A：带水印图纸生成

##### 现有基础

`utils/exportCanvas.js` 已经有完整的图纸生成逻辑：

- 已有 `cellSize` 动态计算

| 存储 | 生成的水印图纸上传到 OSS/云存储，URL 存入 artworks 表的 `cover_url` |

##### 数据流

```

  ↓

  ↓

  ↓

```
